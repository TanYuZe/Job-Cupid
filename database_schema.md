# Job-Cupid: Database Schema

## Design Principles
- All primary keys are `UUID` (type 4) generated by the application, never by the database — enables horizontal sharding and avoids sequential ID enumeration attacks
- All tables include `created_at` and `updated_at` timestamps managed by JPA `@CreationTimestamp` / `@UpdateTimestamp`
- Soft deletes on users and jobs via `deleted_at TIMESTAMPTZ NULL` (never hard-delete business records)
- All foreign keys have explicit indexes beyond the constraint itself where queries filter on them

## Tables

### 1. `users`

Stores all user accounts regardless of role. Role determines which features and related tables apply.

```sql
CREATE TABLE users (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    email               VARCHAR(255)    NOT NULL,
    password_hash       VARCHAR(255)    NOT NULL,
    first_name          VARCHAR(100)    NOT NULL,
    last_name           VARCHAR(100)    NOT NULL,
    role                VARCHAR(20)     NOT NULL CHECK (role IN ('USER', 'EMPLOYER', 'ADMIN')),
    is_premium          BOOLEAN         NOT NULL DEFAULT FALSE,
    bio                 TEXT,
    location            VARCHAR(255),
    photo_url           VARCHAR(500),
    is_active           BOOLEAN         NOT NULL DEFAULT TRUE,
    is_banned           BOOLEAN         NOT NULL DEFAULT FALSE,
    last_login_at       TIMESTAMPTZ,
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    deleted_at          TIMESTAMPTZ     NULL
);

-- Constraints
ALTER TABLE users ADD CONSTRAINT uq_users_email UNIQUE (email);

-- Indexes
CREATE INDEX idx_users_email       ON users (email) WHERE deleted_at IS NULL;
CREATE INDEX idx_users_role        ON users (role)  WHERE deleted_at IS NULL;
CREATE INDEX idx_users_is_active   ON users (is_active, deleted_at);
```

**Rationale:**
- Single table for all roles avoids costly joins for auth checks
- `is_premium` is denormalized here for fast entitlement checks; the source of truth is the `subscriptions` table, reconciled by a scheduled job
- `deleted_at` pattern allows data recovery and preserves referential integrity

### 2. `candidate_profiles`

Extended profile data for candidates (role = USER). One-to-one with `users`.

```sql
CREATE TABLE candidate_profiles (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID            NOT NULL,
    headline            VARCHAR(255),
    resume_url          VARCHAR(500),
    skills              TEXT[],                         -- PostgreSQL array for skill tags
    years_of_experience SMALLINT        CHECK (years_of_experience >= 0),
    desired_salary_min  INTEGER,
    desired_salary_max  INTEGER,
    preferred_remote    BOOLEAN         NOT NULL DEFAULT FALSE,
    preferred_location  VARCHAR(255),
    is_open_to_work     BOOLEAN         NOT NULL DEFAULT TRUE,
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_cp_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT uq_cp_user UNIQUE (user_id),
    CONSTRAINT chk_salary_range CHECK (
        desired_salary_min IS NULL OR desired_salary_max IS NULL
        OR desired_salary_min <= desired_salary_max
    )
);

-- Indexes
CREATE INDEX idx_cp_user_id         ON candidate_profiles (user_id);
CREATE INDEX idx_cp_skills          ON candidate_profiles USING GIN (skills);
CREATE INDEX idx_cp_open_to_work    ON candidate_profiles (is_open_to_work);
```

### 4. `jobs`

Job postings created by employers.

```sql
CREATE TABLE jobs (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    employer_id         UUID            NOT NULL,
    title               VARCHAR(255)    NOT NULL,
    description         TEXT            NOT NULL,
    category            VARCHAR(100)    NOT NULL,
    location            VARCHAR(255),
    is_remote           BOOLEAN         NOT NULL DEFAULT FALSE,
    salary_min          INTEGER,
    salary_max          INTEGER,
    currency            CHAR(3)         NOT NULL DEFAULT 'USD',
    employment_type     VARCHAR(50)     NOT NULL CHECK (employment_type IN ('FULL_TIME', 'PART_TIME', 'CONTRACT', 'INTERNSHIP', 'FREELANCE')),
    experience_level    VARCHAR(50)     CHECK (experience_level IN ('ENTRY', 'MID', 'SENIOR', 'LEAD', 'EXECUTIVE')),
    required_skills     TEXT[],
    status              VARCHAR(20)     NOT NULL DEFAULT 'ACTIVE' CHECK (status IN ('ACTIVE', 'PAUSED', 'CLOSED', 'ARCHIVED')),
    boost_score         INTEGER         NOT NULL DEFAULT 0,          -- Premium: elevated to 100
    application_count   INTEGER         NOT NULL DEFAULT 0,          -- Denormalized counter
    search_vector       TSVECTOR,                                    -- Full-text search
    expires_at          TIMESTAMPTZ,
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    deleted_at          TIMESTAMPTZ     NULL,

    CONSTRAINT fk_jobs_employer FOREIGN KEY (employer_id) REFERENCES users(id),
    CONSTRAINT chk_jobs_salary CHECK (
        salary_min IS NULL OR salary_max IS NULL
        OR salary_min <= salary_max
    )
);

-- Indexes
CREATE INDEX idx_jobs_employer_id   ON jobs (employer_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_jobs_status        ON jobs (status)      WHERE deleted_at IS NULL;
CREATE INDEX idx_jobs_category      ON jobs (category)    WHERE status = 'ACTIVE' AND deleted_at IS NULL;
CREATE INDEX idx_jobs_location      ON jobs (location)    WHERE status = 'ACTIVE' AND deleted_at IS NULL;
CREATE INDEX idx_jobs_remote        ON jobs (is_remote)   WHERE status = 'ACTIVE';
CREATE INDEX idx_jobs_feed          ON jobs (boost_score DESC, created_at DESC, id DESC) WHERE status = 'ACTIVE' AND deleted_at IS NULL;
CREATE INDEX idx_jobs_search        ON jobs USING GIN (search_vector);
CREATE INDEX idx_jobs_skills        ON jobs USING GIN (required_skills);
CREATE INDEX idx_jobs_salary        ON jobs (salary_min, salary_max) WHERE status = 'ACTIVE';

-- Trigger to maintain search_vector
CREATE OR REPLACE FUNCTION jobs_search_vector_update() RETURNS TRIGGER AS $$
BEGIN
    NEW.search_vector :=
        setweight(to_tsvector('english', COALESCE(NEW.title, '')), 'A') ||
        setweight(to_tsvector('english', COALESCE(NEW.category, '')), 'B') ||
        setweight(to_tsvector('english', COALESCE(NEW.description, '')), 'C');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trig_jobs_search_vector
    BEFORE INSERT OR UPDATE ON jobs
    FOR EACH ROW EXECUTE FUNCTION jobs_search_vector_update();
```

**Rationale:**
- `boost_score` denormalized for fast feed sorting; updated by subscription service and scheduled reset job
- `application_count` counter updated atomically via `UPDATE jobs SET application_count = application_count + 1` to avoid N+1 aggregate queries
- Composite feed index `(boost_score DESC, created_at DESC, id)` enables efficient keyset pagination

### 5. `candidate_swipes`

Records every swipe a candidate makes on a job.

```sql
CREATE TABLE candidate_swipes (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    candidate_id        UUID            NOT NULL,
    job_id              UUID            NOT NULL,
    action              VARCHAR(10)     NOT NULL CHECK (action IN ('LIKE', 'PASS')),
    swiped_at           TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_cs_candidate FOREIGN KEY (candidate_id) REFERENCES users(id),
    CONSTRAINT fk_cs_job       FOREIGN KEY (job_id)       REFERENCES jobs(id),
    CONSTRAINT uq_cs_candidate_job UNIQUE (candidate_id, job_id)   -- One swipe per (candidate, job)
);

-- Indexes
CREATE INDEX idx_cs_candidate_id    ON candidate_swipes (candidate_id);
CREATE INDEX idx_cs_job_id          ON candidate_swipes (job_id);
CREATE INDEX idx_cs_action_like     ON candidate_swipes (candidate_id, job_id) WHERE action = 'LIKE';
```

**Rationale:**
- `UNIQUE (candidate_id, job_id)` enforces idempotency at DB level; application layer uses `ON CONFLICT DO UPDATE SET action = EXCLUDED.action` to handle re-swipes
- Partial index on `action = 'LIKE'` accelerates the match gate query

### 6. `applications`

Tracks candidate applications to jobs. Bridges candidates and jobs with status.

```sql
CREATE TABLE applications (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    candidate_id        UUID            NOT NULL,
    job_id              UUID            NOT NULL,
    cover_letter        TEXT,
    resume_url          VARCHAR(500),                   -- Snapshot of resume at time of apply
    status              VARCHAR(20)     NOT NULL DEFAULT 'PENDING'
                                        CHECK (status IN ('PENDING', 'REVIEWED', 'SHORTLISTED', 'ACCEPTED', 'REJECTED', 'WITHDRAWN')),
    applied_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    reviewed_at         TIMESTAMPTZ,
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_app_candidate FOREIGN KEY (candidate_id) REFERENCES users(id),
    CONSTRAINT fk_app_job       FOREIGN KEY (job_id)       REFERENCES jobs(id),
    CONSTRAINT uq_app_candidate_job UNIQUE (candidate_id, job_id)  -- One application per (candidate, job)
);

-- Indexes
CREATE INDEX idx_app_candidate_id   ON applications (candidate_id);
CREATE INDEX idx_app_job_id         ON applications (job_id);
CREATE INDEX idx_app_status         ON applications (job_id, status);
CREATE INDEX idx_app_applied_at     ON applications (job_id, applied_at DESC);
```

### 7. `employer_swipes`

Records every swipe an employer makes on an applicant (must be an existing application).

```sql
CREATE TABLE employer_swipes (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    employer_id         UUID            NOT NULL,
    application_id      UUID            NOT NULL,
    candidate_id        UUID            NOT NULL,         -- Denormalized for match gate query efficiency
    job_id              UUID            NOT NULL,         -- Denormalized for match gate query efficiency
    action              VARCHAR(10)     NOT NULL CHECK (action IN ('LIKE', 'REJECT')),
    swiped_at           TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_es_employer     FOREIGN KEY (employer_id)    REFERENCES users(id),
    CONSTRAINT fk_es_application  FOREIGN KEY (application_id) REFERENCES applications(id),
    CONSTRAINT fk_es_candidate    FOREIGN KEY (candidate_id)   REFERENCES users(id),
    CONSTRAINT fk_es_job          FOREIGN KEY (job_id)         REFERENCES jobs(id),
    CONSTRAINT uq_es_employer_app UNIQUE (employer_id, application_id)
);

-- Indexes
CREATE INDEX idx_es_employer_id     ON employer_swipes (employer_id);
CREATE INDEX idx_es_application_id  ON employer_swipes (application_id);
CREATE INDEX idx_es_match_gate      ON employer_swipes (candidate_id, job_id) WHERE action = 'LIKE';
```

**Rationale:**
- `candidate_id` and `job_id` are denormalized to avoid a join with `applications` on every match gate evaluation
- Partial index on `action = 'LIKE'` makes the match gate query a simple index scan

### 8. `matches`

Immutable record of completed three-condition matches.

```sql
CREATE TABLE matches (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    candidate_id        UUID            NOT NULL,
    employer_id         UUID            NOT NULL,
    job_id              UUID            NOT NULL,
    application_id      UUID            NOT NULL,
    status              VARCHAR(20)     NOT NULL DEFAULT 'ACTIVE'
                                        CHECK (status IN ('ACTIVE', 'ARCHIVED', 'EXPIRED')),
    matched_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_match_candidate   FOREIGN KEY (candidate_id)   REFERENCES users(id),
    CONSTRAINT fk_match_employer    FOREIGN KEY (employer_id)    REFERENCES users(id),
    CONSTRAINT fk_match_job         FOREIGN KEY (job_id)         REFERENCES jobs(id),
    CONSTRAINT fk_match_application FOREIGN KEY (application_id) REFERENCES applications(id),
    CONSTRAINT uq_match_candidate_job UNIQUE (candidate_id, job_id)   -- One match per (candidate, job) ever
);

-- Indexes
CREATE INDEX idx_matches_candidate_id   ON matches (candidate_id, matched_at DESC);
CREATE INDEX idx_matches_employer_id    ON matches (employer_id,  matched_at DESC);
CREATE INDEX idx_matches_job_id         ON matches (job_id);
```

**Rationale:**
- `UNIQUE (candidate_id, job_id)` is the idempotency guard at DB level; Kafka consumer uses `INSERT ... ON CONFLICT DO NOTHING`
- Sorting by `matched_at DESC` for the matches feed is covered by the composite indexes

### 9. `subscriptions`

Premium subscription records. The `is_premium` flag on `users` is a denormalized fast-path; this table is the source of truth.

```sql
CREATE TABLE subscriptions (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID            NOT NULL,
    plan                VARCHAR(50)     NOT NULL CHECK (plan IN ('PREMIUM_MONTHLY', 'PREMIUM_ANNUAL')),
    status              VARCHAR(20)     NOT NULL DEFAULT 'ACTIVE'
                                        CHECK (status IN ('ACTIVE', 'CANCELLED', 'EXPIRED', 'PAST_DUE')),
    stripe_customer_id  VARCHAR(255),
    stripe_subscription_id VARCHAR(255),
    current_period_start TIMESTAMPTZ    NOT NULL,
    current_period_end   TIMESTAMPTZ    NOT NULL,
    cancelled_at        TIMESTAMPTZ,
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_sub_user FOREIGN KEY (user_id) REFERENCES users(id)
);

-- Indexes
CREATE INDEX idx_sub_user_id        ON subscriptions (user_id, status);
CREATE INDEX idx_sub_stripe_sub_id  ON subscriptions (stripe_subscription_id);
CREATE INDEX idx_sub_expiry         ON subscriptions (current_period_end) WHERE status = 'ACTIVE';
```

### 10. `notifications`

In-app notification records. Dispatched externally (email/push) via Kafka.

```sql
CREATE TABLE notifications (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID            NOT NULL,
    type                VARCHAR(50)     NOT NULL
                                        CHECK (type IN ('MATCH_CREATED', 'APPLICATION_RECEIVED',
                                                        'APPLICATION_STATUS_CHANGED', 'SWIPE_RECEIVED',
                                                        'SUBSCRIPTION_RENEWED', 'SUBSCRIPTION_EXPIRING')),
    title               VARCHAR(255)    NOT NULL,
    body                TEXT            NOT NULL,
    reference_id        UUID,                               -- e.g., matchId, applicationId
    reference_type      VARCHAR(50),                        -- e.g., 'MATCH', 'APPLICATION'
    is_read             BOOLEAN         NOT NULL DEFAULT FALSE,
    read_at             TIMESTAMPTZ,
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_notif_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

-- Indexes
CREATE INDEX idx_notif_user_unread  ON notifications (user_id, created_at DESC) WHERE is_read = FALSE;
CREATE INDEX idx_notif_user_all     ON notifications (user_id, created_at DESC);
```

### 11. `refresh_tokens`

Persistent refresh token registry. Redis is primary; this table is the durable fallback for audit and revocation.

```sql
CREATE TABLE refresh_tokens (
    id                  UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID            NOT NULL,
    token_hash          VARCHAR(255)    NOT NULL,           -- SHA-256 of the raw token
    device_info         VARCHAR(500),
    ip_address          INET,
    issued_at           TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    expires_at          TIMESTAMPTZ     NOT NULL,
    revoked_at          TIMESTAMPTZ,
    is_revoked          BOOLEAN         NOT NULL DEFAULT FALSE,

    CONSTRAINT fk_rt_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT uq_rt_token_hash UNIQUE (token_hash)
);

CREATE INDEX idx_rt_user_id     ON refresh_tokens (user_id) WHERE is_revoked = FALSE;
CREATE INDEX idx_rt_token_hash  ON refresh_tokens (token_hash);
CREATE INDEX idx_rt_expiry      ON refresh_tokens (expires_at) WHERE is_revoked = FALSE;
```

## Performance Considerations

- **Keyset pagination** on the job feed uses `(boost_score DESC, created_at DESC, id DESC)` — covered by `idx_jobs_feed`
- **Match gate query** runs on two partial indexes: `idx_cs_action_like` and `idx_es_match_gate` — both index-only scans
- **Full-text search** on jobs uses `TSVECTOR` column + GIN index — updated by trigger, not at query time
- **Swipe exclusion** in the feed query: `WHERE j.id NOT IN (SELECT job_id FROM candidate_swipes WHERE candidate_id = ?)` is replaced with a LEFT JOIN + `IS NULL` pattern for index use
- **application_count** counter uses atomic `UPDATE ... SET application_count = application_count + 1` to avoid aggregate query overhead on employer dashboards
- For high-write tables (`candidate_swipes`, `notifications`), consider table partitioning by `created_at` (range partitioning, monthly) when row count exceeds 100M
